Ruby Arrays and Hashes

March 14, 2015

When I first started learning Ruby on Codecademy, before Phase 0, I thought Arrays and Hashes were two completely separate things. Even as I read blog posts from DBC alumni about how they kept picking the wrong thing, I thought to myself, they're totally different, how could you get them confused? Well, it makes a lot more sense now. As it turns out, Arrays and Hashes in Ruby are very similar. Both are ways of storing and indexing information. Both have keys and values. And both grow as more information is added. But there is one key difference: information in Arrays are organized using an integer as the key, whereas anything can be a key with Hashes (a string, float number, etc.)

To take a step back, when you think of an index, such as one in a textbook, each key word or phrase is assigned to a page number where information can be found about it. This is exactly how information is stored in both arrays and hashes. An array is simply a collection of information organized in a logical sequence. It looks something like this: 
      array = [1,2,3,4,5]
Each number within the brackets is indexed according to the position it's in. So "1" is in the zero-eth position; its key is 0 and the value is 1. "2" is in the first position; its key is 1 and the value is 2, and so on and so forth. Once the array is created you can tell the computer to print the value by inputting the key.

You can have an array full of other types of things, such as this: 
    array_2 = ["Sam",2,"DBC",8.06]
The basic construct of the key and value pairs still exists. If you ask the computer what value is stored in the key 3, the response will be 8.06.

Hashes on the other hand can have non-numeric keys. As such you define both the key and the value. For example: 
    months{
      "January" => 1,
      "February" => 2,
      "March" => 3,
    }
As you can see above, January is now a key and 1 is its value.

In short, while arrays and hashes are very similar methods of storing information, there are definitely use cases for both, and it's important to consider the project you're working on, and the scope of the data, before deciding which method to use.
