Asking Good Questions

April 21, 2015

One of the main things I've learned during Phase 0 has been that google is your best friend when learning to code. I've found that I always have questions that aren't addressed in the codecademy courses or in the documentation, and it's important to be able to find those answers on google. Outside of posting a question on stack overflow or a similar website, it's important to be able to phrase a question on a search engine to be able to get the best website answers possible.

I think question-writing is very similar to writing/editing in general as it's best practice to make sure everything is spelled correctly, your grammar is accurate, and you're including all necessary information. In a coding context, it's important to include what language you're working in, and possibly an example of your code (as long as you aren't posting a company's private code) so that someone can review it and see what's going on. I think it can also be beneficial to specify what type of computer you're using, especially for help with using the terminal or installing software.

The biggest hurdle I've had in finding an answer during Phase 0 was not knowing how to phrase a question correctly. For example, on a recent challenge we had to create a nested for loop through an object in JavaScript, but all answers kept being about iterating through nested arrays. It was a bit frustrating trying to phrase my question specifically enough to find a good resource without typing out the exact situation.

I think being able to ask good questions in person is just as important, if not more so, than being able to successfully find resources online. I've found through my work experience that people's time is the most valuable resource. An important thing to keep in mind in real-life situations is to first sit down and think about the question that you want to ask. Have you asked this before? Do you have any existing notes or resources you can utilize first? If not, it's important to be as concise and clear as possible. Start with a high level background describing the circumstance and then ask the specific question you had in mind. If you start with the question, another person might have no clue what you're talking about and then you have to back up and give the broader overview anyways. After you ask the question, I think it's also a good habit to write a note for yourself on what the response was so that you have something to reference later on. You always think you'll remember the answer later, but I've found that's generally not the case. It's better to refer to your own notes later, rather than asking the same question twice.

Building My Website

April 22, 2015

This week I decided to purchase a domain name for my website and try to set it up - this wound up being much more complicated than I thought it would be (and I think it should be), but with some help from my dad, I got it working, and thought I'd write about the process. This post will be solely about linking a GitHub pages site with a custom domain name. You can see the GitHub FAQ page from which I started. First you need to purchase a domain name (URL) through a DNS (domain name system) provider like GoDaddy - you can find deals for around $0.99 for the first year. In this case, my site will still be hosted by GitHub, so I didn't pay for hosting services, just the URL.

Next you need to add a CNAME file to your GitHub repo - you can do this by creating a new file in Sublime (or any text editor) and saving it as CNAME in your root directory of your repo. The only text in the file should be your URL that you purchased (in my case www.samanthasarah.com). By using a CNAME file, you are using the subdomain, so you have to include the "www." at the beginning. Once this has been committed, you should be able to click on your repo's settings and it should say something along the lines of "Your site is published at http://www.samanthasarah.com". This is needed in order to link your personal GitHub page with your URL. In the next step you are only linking your domain name to GitHub's IP address, and this file provides the link to your specific site.

Now comes the tricky part - you have to set this up on the DNS provider's side as well. I purchased my URL through GoDaddy, so I went to their website to login and manage my domain there. Under your control panel on their site you should be able to add/edit your DNS Zones. Under the CNAME section, you add a record with the Host of www and it Points To your github.io page. By doing this, when you type www.samanthasarah.com in the browser, your github.io site will load. You could feasibly stop here, but you wouldn't be able to get to your site if a user typed anything but that exact URL (for example samanthasarah.com wouldn't work). In order for all cases to work we have to set-up an "A" record which is also managed in the same area of your DNS provider. The Host in this case should be @ and it should Point To the IP address of your host (in this case GitHub) - I had to search through their FAQ to find this, but it's located here. Once you set-up both of these records, a user can get to your website by either including or excluding the "www." portion of your URL.

I hope that was helpful, and good luck setting up your own custom URLs for your personal site!

Recursion

April 16, 2015

Recursion is a programming function in which you have self-referential definitions. What this means is that the method or function calls itself. You can think of recursion like a loop, but instead of writing an each, for, or while loop, the method just keeps calling itself until the defintion tells it to stop. This is an important aspect of using recursion, because if you don't include what's called a base case, the method will run indefinitely (or until your program crashes).

Here is a basic example of a method using recursion: 
      def greeting(n)
        if n<=0
          p "Bye!"
        else
          p "Hi!"
          greeting(n-1)
        end
      end
If you then call this method by writing: 
      greeting(3)
your output would be: 
      "Hi!"
      "Hi!"
      "Hi!"
      "Bye!"
What this does it you define your base case as n<=0, where the method will print "Bye" and the program ends. Otherwise it calls the method again using n-1. So in this example the first time the program runs, n=3 so it prints "Hi!" and runs the method again with n=2. This also prints "Hi!" and runs the method again with n=1. Yet again it prints "Hi!" and then runs the method with n=0. In this case we hit our base case so the method just prints "Bye!" and ends.

Methods with recursion can be very powerful, such as creating a method to create factorials or add all the numbers in a range. Instead of having to iterate, you can use recursion and it will keep adding to a running total. For large tasks, recursion can generally be written with less code than iteration, and can be cleaner. However with smaller tasks, iteration can be better to use as you don't have to worry about the call running indefinitely.

JavaScript vs. Ruby

April 8, 2015

This week we got our first glimpse into JavaScript. Objectively speaking, the language is pretty similar to Ruby, and most of the concepts that exist in Ruby also exist in JavaScript, albeit with different syntax. However, as we dug deeper it became more and more clear that JavaScript just doesn't have as many built-in capabilities as Ruby does. I found myself wanting to do something that was so easy in Ruby (like calling a built-in method), and instead having to write the function in JavaScript first and then call it. I remember back to my first week with Ruby and I felt lost because there was so much you could do, it felt infinite. I remember longing for the structure of SQL that I knew, but now that I'm playing in JavaScript, part of me keeps longing to go back and use Ruby instead! I do think that learning Ruby first was extremely helpful though, as catching onto JavaScript has been significantly easier since the underlying principles are very similar.

While similar, the syntax and structure of JavaScript does have some major differences. For example, iterating (looping) through data structures not the same. Whereas Ruby has many built-in methods that can tell how long your data structure is, and go through each instance automatically, JavaScript does not, and you have to tell the code how long you want the loop to go for. Also whereas Ruby has several different ways of going about this (see previous post on Enumerables), JavaScript really has 2 - while statements, and for statements.

Let's go through both examples. First we'll define an array that we want to iterate through, for example: 
      var array = ["1","2","3","4","5"]
A while statement for this array would look like this: 
      var counter = 0;
      while (counter < array.length){
        console.log(array[counter]);
        counter ++;
      }
Going through the code, we first define a variable called counter and set it equal to 0. The text in the parentheses tell the loop how many time to execute. In this case we want the loop to execute the length of the array - 1. So here the length of the array is 5, and the loop will actually run 5 times since we are starting the variable at 0. So it will run when counter is 0, 1, 2, 3, and 4 in order to go through each element of the array (remember the index of the array begins at 0, not 1). The code in the curly brackets then is the code that should be executed. So we are telling the computer, for every loop, print out the element of the array to the console, and add 1 to the counter. If we don't include the last part, the code will just print out the first element of the array to infinity (or until it crashes).

For statements are very similar, but it allows you to write everything on one line instead of breaking it out. So the same example as above but in a for loop would look like this: 
      for (var counter = 0; counter < array.length; counter ++){
        console.log(array[counter]);
      }
Again, to go through the code, the text in the parentheses are just setting the stage for how you want the loop to run, and the text in the curly bracket is the code to be executed each loop. The elements in the parantheses are separated by semi-colons: the first element sets our counter variable to 0, the second element tells the loop to run the length of the array - 1 times, and the last element adds 1 to the counter variable on each run. The code to be executed is just printing out to the console each element of the array.

To illustrate the difference between this and how Ruby executes loops, let's look at the same example in Ruby. We have our array: 
      array = ["1","2","3","4","5"]
Now, instead of having to write all that code like in JavaScript, we can use an easy built-in method like this: 
      array.each {|element| puts element}
With Ruby you don't have to explicitly tell the loop how long to execute for, it will automatically go through every element in the array given. So as mentioned above, while similar, the languages have different capabilities and built-in methods, as well as slightly different syntax.

Blocs, Procs, and Lambdas

March 31, 2015

Blocs, Procs, and Lambdas - does it sound like we're back in a Frat house in college yet? As opposed to Greek letters, Blocs, Procs, and Lambdas are actually important tools that you can use in Ruby code. They are essentially snippets of code which you tell the computer to do within a method. We've already seen them when using enumerables, but here is an example: 
    def method(arguments)
      arguments.each {|argument| do something to the argument}
    end
The little bit in the curly brackets {} is what we call a Bloc. Blocs function like a method in that they are telling the code to do something, but they are technically undefined and unnamed.

Procs are very similar to Blocs in that they also are bits of code which tell the computer to do something, however they are different in that they are defined and saved as a variable. So you can take the exact same code as in our example above and define it as a variable which then can get passed into the method, instead of writing it out: 
    proc = Proc.new {|argument| do something to the argument}

    def method(arguments)
      arguments.each(&proc)
    end
The only difference here is that you use the "&" sign when you pull the Proc into your method. These can be useful when you want to assign your Proc more than one thing to do, or if you want to refer back to it multiple times in your code.

Last, but not least, we have Lambdas. Lambdas are rarely used, but they are the same thing as Procs, but are more detailed like methods. The other main difference is that you can use the "return" keyword within a Lambda since it will only return the Lambda, not the entire method. If you use "return" in a Bloc or a Proc, it will exit out of the entire method at the return. The other main difference is that Lambdas will check the number of arguments passed into it (like methods), whereas Procs and Blocs do not.

Ruby Classes

March 29, 2015

When working with Ruby, methods are the basic code which tells the computer to do something. This capability is great, but what happens when you are routinely running multiple methods, and can't easily find the code you need? Classes are a functionality in Ruby which allows you to group all the methods that are applicable to certain types of objects. You can then run an instance of the Class, and all the methods are there for you. It saves time and confusion by bucketing them all together. Let's run through an example: in finance, an asset manager needs to keep track of, and be able to quickly access, multiple data points relating to any one of their portfolios. Let's build a class which will allow us to do that. We first need to initialize the class by parsing in the portfolio's name and perhaps a short code for ease of reference, along with a date that you would want the output to be as-of: 
    class PortfolioInfo

      def initialize (name, short_code, date)
      @name = name
      @short_code = short_code
      @date = date
      end

      #other code here to query underlying databases

    end
Now we can write however many methods where the comments are which will take either the full portfolio name, or the short code, and use it to query underlying databases to gather information on the account as of the specified date - for example key metrics (P/E, Average Market Cap, Turnover), or generate a list of holdings as of that date. This saves us from having to write new methods every time we want to access a piece of data. We can also run this for as many portfolios and dates as we want to.

The information that we initially parse into the instance becomes what are called instance variables - they are defined at the top with the "@" sign in front of them, and this allows them to be used and referenced throughout the other methods you build within the class.

Ruby Enumerables

March 16, 2015

Enumerables. What the heck does that word even mean? When first reading up on Enumerables, we were supposed to research certain Enumerable methods like map, group_by, and cycle. However, when looking up map in the Ruby docs, I was directed towards the section on Arrays, which confused me even more. Is map a method used on Arrays, or a method under Enumerables? As it turns out, it's really both. An Enumerable is simply a collection of methods which can be used to search or manipulate collections of data (therefore they can be used to search or manipulate Arrays as an example). Therefore to break it down even further, map is a method belonging to a class of methods called Enumerables, which can be used on collections of data such as Arrays and Hashes. Make sense?

There are a few different types of Enumerable methods which do different things:
  • Methods which don't actually touch the underlying data and just go through each item
    • each
  • Methods that pull certain elements out of the underlying data based on criteria and populate a new collection with them
    • select
  • Methods which change all underlying elements through a block and populate a new collection with them. Note this does not alter the original collection.
    • collect
    • map
  • Methods which actually change the original collection of data (are destructive).
    • map!
  • Methods which take a collection of underlying data and reduces it to a single value (i.e. adding up all elements).
    • inject

Getting a little off topic, the original purpose of this post was to discuss one Enumerable, of which I chose map. Map essentially creates a new collection of data (let's call it an Array going forward for ease), using all the original elements of an array, and changes them using a block. The syntax looks like this: 
      array.map {|element| element+2}
What this says is take each element of the original array, add 2 to it, and populate the new value in a new array. So if the original array was [1,2,3,4,5], the new array would look like [3,4,5,6,7]. This is helpful as you can use this method either destructively (using map!), or non-destructively using map.

map can also be used with Hashes, but you must include both the key and value in the code. For example: 
      hash.map {|key, value| value+2}

Ruby Arrays and Hashes

March 14, 2015

When I first started learning Ruby on Codecademy, before Phase 0, I thought Arrays and Hashes were two completely separate things. Even as I read blog posts from DBC alumni about how they kept picking the wrong thing, I thought to myself, they're totally different, how could you get them confused? Well, it makes a lot more sense now. As it turns out, Arrays and Hashes in Ruby are very similar. Both are ways of storing and indexing information. Both have keys and values. And both grow as more information is added. But there is one key difference: information in Arrays are organized using an integer as the key, whereas anything can be a key with Hashes (a string, float number, etc.)

To take a step back, when you think of an index, such as one in a textbook, each key word or phrase is assigned to a page number where information can be found about it. This is exactly how information is stored in both arrays and hashes. An array is simply a collection of information organized in a logical sequence. It looks something like this: 
      array = [1,2,3,4,5]
Each number within the brackets is indexed according to the position it's in. So "1" is in the zero-eth position; its key is 0 and the value is 1. "2" is in the first position; its key is 1 and the value is 2, and so on and so forth. Once the array is created you can tell the computer to print the value by inputting the key.

You can have an array full of other types of things, such as this: 
    array_2 = ["Sam",2,"DBC",8.06]
The basic construct of the key and value pairs still exists. If you ask the computer what value is stored in the key 3, the response will be 8.06.

Hashes on the other hand can have non-numeric keys. As such you define both the key and the value. For example: 
    months{
      "January" => 1,
      "February" => 2,
      "March" => 3,
    }
As you can see above, January is now a key and 1 is its value.

In short, while arrays and hashes are very similar methods of storing information, there are definitely use cases for both, and it's important to consider the project you're working on, and the scope of the data, before deciding which method to use.

Git Bloggin'

February 25, 2015

As a completely new programmer myself, I am writing this blog in hopes of tracking my own thoughts as well as potentially educating others who are interested in learning about programming. This post is my first technical blog and will cover the basics of git: version control, how git tracks changes, and the benefits of using GitHub to store code.

First things first: what is git? Git is a tool that you can download to your computer which makes it easy to track changes to certain documents and merge changes. It is very similar in concept to how the tracking changes functionality works in Microsoft Word. Not only can you use it alone to track your own changes and update files, but it also allows multiple people to work on the same file and make changes which can then be merged to produce a final version. Git uses something called version control to do this - essentially it saves a separate version of a file every time you tell it to. This can be useful if you make an error in a file and want to revert to how the file was at an earlier point in time. The more times you save your changes, the more possibilities exist for this. Developers use version control to be able to revert to prior code when something in the new code fails. Oftentimes code is crucial to business operations and it is imperative to be able to revert to the tested, working code in order to get a project done. Version control allows the developer to re-deploy the working code, and then go back and make the necessary updates to new code.

GitHub is an online storage system for code. Developers use GitHub to remotely store their code, so others can access it and make changes of their own. The benefits of this is that you can have multiple people making changes to the same document housed on GitHub and the system will keep track of these and merge them at the end to form one coherent document. The individuals don't have to be in the same location to collaborate on a project.
